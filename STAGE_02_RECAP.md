## Data formats actually used (Stage 3 depends on these)
- **Objective spec shape used** (real examples, exactly as loaded):
  - `{ "id":"coffee", "type":"interact", "target":"coffee_pot", "text":"Put the coffee out" }`
  - `{ "id":"file",   "type":"deliver",  "item":"folder", "target":"desk", "text":"File the folder" }`
- **Entity/prop spec shape used** (real examples):
  - `{ "type":"prop", "id":"coffee_pot", "sprite":"coffee_pot", "x":64,  "y":64, "verb":"use" }`
  - `{ "type":"prop", "id":"folder",     "sprite":"folder",     "x":120, "y":96, "verb":"pickup", "item":"folder" }`
  - `{ "type":"prop", "id":"desk",       "sprite":"desk",       "x":200, "y":64, "verb":"accepts", "accepts":"folder" }`
  - Scene config wrapper: `{ spawn:{x,y}, entities:[...], objectives:[...] }` (spawn is in pixels).
- **Event names the tracker listens for:** `interact:done { id }` and `deliver:done { item, target }`.
  - Also emitted by the verb layer (helpers, not required by the tracker): `pickup:done { item }`, `drop:done { item }`.
  - The tracker also emits: `objective:updated (arrayOfObjectives)` (HUD listens) and `objectives:allcomplete` (+ calls `onComplete`).
  - The bus is a dedicated `Phaser.Events.EventEmitter` created in `PeriodScene` (`scene.bus`), kept separate from Phaser's built-in `scene.events`.
- **Interact key(s) + interact range constant:** interact = `E` and `Enter`; `INTERACT_RANGE = 22` (px), a named export at the top of `interaction.js`.
- **Drop key:** `Q`. (Now taken. `Space` remains free/reserved.)
- **Carry representation:** a single field `heldItem` (the item id string, e.g. `'folder'`) plus a `heldSprite` image drawn 16px above the player's head. The item id doubles as the texture key. Delivering or … clearing the slot destroys the held sprite and sets `heldItem = null`.

## Deviations from canon (and why)
- **Carry lives inside `interaction.js`**, not a separate file — canon says "verbs in interaction.js", and pickup/drop/deliver are verbs. Kept as a clearly separated Carry section.
- **Props have a static physics body but no collider this stage** — you can stand on a prop to interact, which is simplest for now. If Stage 3 wants solid furniture, add `this.physics.add.collider(player, prop)`.
- Nothing else diverged; all out-of-scope systems (tasks.js, named task types, period pools, dialogue, hearts/score, detention slip, Hall Duty, Lunch) were intentionally NOT built.

## Friction points (to improve future starter docs)
- `interaction.js` imports Phaser, so it can't be unit-tested with a bare `node -e import` (no DOM). I verified it via `node --check` (syntax) + a headless-browser screenshot instead. `objectives.js` is Phaser-free and *was* Node-tested directly — worth keeping future core systems Phaser-free where possible so they stay unit-testable.
- The brief's completion path (`onComplete`) and the event names were clear and matched cleanly; no ambiguity there.

## For the owner — how to review and ship

**1) Test locally** (from inside `other-duties/`):
```
npm run dev
```
Then open the printed http://localhost:5173/ , press any key at the title, and in the room:
- walk to the **coffee pot** (top-left) and press **E** → "Put the coffee out" ticks off;
- walk onto the **folder** and press **E** to pick it up (it rides above your head), carry it to the **desk** and press **E** → "File the folder" ticks off;
- when both are done you'll see **"All duties complete!"**. (Drop the folder anytime with **Q**.)

**2) Push + deploy** (this is what makes it live):
```
git push origin main
```
Vercel auto-builds on push; about a minute later **https://other-duties.vercel.app/** updates.

**3) What you'll see once it's live:** the same interactive sandbox above, running on the public URL instead of just your machine.
